Problema detectado:

Los valores de `speedup` para OpenMP (2386, 5555, 10411, 14813, etc.) no representan speedup paralelo.

Parece que actualmente se está calculando algo similar a:

speedup = candidates_per_second_openmp / candidates_per_second_python

o

speedup = tiempo_python / tiempo_openmp

lo cual mide la diferencia entre implementaciones (Python vs C), pero NO el speedup paralelo de OpenMP.

Definiciones correctas:

Para cada implementación, el speedup debe calcularse respecto a la versión base de ESA MISMA implementación.

OpenMP:

speedup_openmp(p) = T_openmp_1 / T_openmp_p

donde:

- T_openmp_1 = tiempo de OpenMP con 1 hilo
- T_openmp_p = tiempo de OpenMP con p hilos

Ejemplo:

T_openmp_1 = 0.0341 s
T_openmp_2 = 0.0150 s
T_openmp_4 = 0.0082 s
T_openmp_8 = 0.0055 s

Entonces:

speedup(1) = 1.0
speedup(2) = 0.0341 / 0.0150 ≈ 2.27
speedup(4) = 0.0341 / 0.0082 ≈ 4.17
speedup(8) = 0.0341 / 0.0055 ≈ 6.18

La eficiencia debe calcularse como:

efficiency = speedup / workers

Por ejemplo:

efficiency(4) = 4.17 / 4 ≈ 1.04

Para Python multicore:

speedup_python = T_python_secuencial / T_python_multicore

efficiency_python = speedup_python / workers

### respuesta

## 1. Dónde se calculaban `speedup`/`efficiency`

- Python: `code/python/common.py: compute_metrics(K, time_s, workers, t_seq)` —
  `speedup = t_seq / time_s`, `efficiency = speedup / workers`, llamado desde
  `sequential.py`/`multicore.py` con `t_seq = read_sequential_time(...)` (la fila "Python
  secuencial" del CSV).
- C OpenMP: `code/C_OpenMP_MPI/scoring_openmp.c: main()` — `read_sequential_time(BENCHMARK_CSV,
  n_items, k_candidates, &t_seq)` leía la misma fila "Python secuencial" y calculaba
  `speedup = t_seq / time_seconds`, `efficiency = speedup / threads`, escritos por
  `append_benchmark`.

## 2. Cuál era el error original

Ambas implementaciones usaban **el mismo baseline** (`t_seq` = tiempo de "Python secuencial")
para `speedup`, independientemente de la implementación:

- Para **Python multicore**, esto es correcto: su baseline P=1 *es* "Python secuencial"
  (misma implementación, mismo lenguaje), así que `speedup = T_python_secuencial /
  T_python_multicore` mide paralelismo real.
- Para **C OpenMP**, esto es incorrecto: "Python secuencial" NO es la versión de 1 hilo de
  "C OpenMP" — es una implementación distinta, en otro lenguaje, sin NumPy/sklearn vectorizado
  vs. un binario `-O2`. Dividir `T_python_secuencial / T_openmp(P)` mezcla dos efectos
  completamente distintos en un solo número:
  1. La diferencia de rendimiento **entre lenguajes/implementaciones** (Python+NumPy+sklearn
     vs. C compilado con `-O2`), que es de **órdenes de magnitud** y no tiene nada que ver con
     paralelismo.
  2. El **speedup paralelo real** de OpenMP al pasar de 1 a P hilos, que para N=50 (un cómputo
     por candidato muy pequeño) es **sublineal** y del orden de unidades (1×–6×).

Al sumarlos en una sola fracción, el resultado (miles) no es interpretable como ninguna de las
dos cosas por separado. Peor aún, `efficiency = speedup/threads` superaba 1.0 por **miles**
(p.ej. 2777× con P=2), lo cual no tiene sentido como "eficiencia de paralelización" (que por
definición está acotada por ≈1.0 en el caso ideal).

## 3. Por qué los valores de miles no representan speedup paralelo

Concretamente, con los datos originales:

```
t_seq (Python secuencial) = 80.97405759998946 s
T_openmp(P=1) = 0.033935772 s   →  80.974/0.033936 = 2386.10
T_openmp(P=2) = 0.014576411 s   →  80.974/0.014576 = 5555.14
T_openmp(P=4) = 0.007777121 s   →  80.974/0.007777 = 10411.83
T_openmp(P=8) = 0.005466170 s   →  80.974/0.005466 = 14813.67
```

Si `2386.10` fuera el "speedup de OpenMP con 1 hilo", eso implicaría que 1 hilo de C es **2386
veces más rápido que sí mismo** — una contradicción (P=1 debe dar speedup=1 por definición). El
número real que se está midiendo ahí es "cuántas veces más rápido es C que Python para este
cómputo", que es una comparación de **lenguajes**, no de **paralelismo**. La progresión
2386→5555→10412→14814 tampoco refleja un patrón de speedup paralelo razonable (seguiría
creciendo más allá de P=número de núcleos, sin límite de Amdahl visible), porque el numerador
(`t_seq` de Python) es constante y el denominador simplemente decrece con P — es la curva de
`1/T_openmp(P)` escalada por una constante enorme, no una curva de speedup.

## 4. Fórmula antigua vs. fórmula nueva

| | Antes (DEC-11, incorrecto para C OpenMP) | Ahora (DEC-13) |
|---|---|---|
| `speedup` (C OpenMP) | `T_python_secuencial / T_openmp(P)` | `T_openmp(P=1) / T_openmp(P)` |
| `efficiency` (C OpenMP) | `speedup / threads` (heredaba la magnitud de miles) | `speedup / threads` (acotado ≈ 1.0) |
| `speedup` (Python multicore) | `T_python_secuencial / T_python_multicore` | sin cambio (ya era correcto: su baseline P=1 es Python secuencial) |
| Comparación C vs Python | ocupaba indebidamente la columna `speedup` | columna nueva `speedup_vs_python = T_python_secuencial / T_impl(P)` |
| Columnas en `benchmark.csv` | 9 | 10 (`+ speedup_vs_python`) |

Implementación:
- `code/python/common.py: compute_metrics(K, time_s, workers, t_self_base=None,
  t_python_seq=None)` — `speedup = t_self_base/time_s` (si hay baseline propio),
  `speedup_vs_python = t_python_seq/time_s` (columna separada, siempre que haya dato de Python
  secuencial).
- `code/C_OpenMP_MPI/scoring_openmp.c` — nueva función `read_last_time(csv, implementation,
  n_items, k_candidates, required_workers, &out)`, generalización de la antigua
  `read_sequential_time`: para `speedup`/`efficiency` busca la fila `"C OpenMP"` con
  `workers=1`; para `speedup_vs_python` busca la fila `"Python secuencial"` (sin filtrar por
  `workers`). `append_benchmark` ahora acepta ambos pares de valores (con flags `has_speedup`/
  `has_vs_python` para dejarlos vacíos si no hay baseline aún, p.ej. al correr P=1 antes que
  exista una fila previa de "C OpenMP").
